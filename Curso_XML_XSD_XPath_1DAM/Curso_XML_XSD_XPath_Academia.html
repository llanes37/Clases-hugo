Mapa del XML

Antes de lanzar XPath, lo primero es entender la forma del documento. Aqui tienes el dataset real traducido a un mapa visual.

Root version xsi:noNamespaceSchemaLocation

biblioteca

Es el contenedor principal. Dentro viven tres zonas: libros, socios y prestamos. El XSD espera este orden exacto porque usa xs:sequence.

Jerarquia principal
<biblioteca version="1.0">   <libros>...</libros>   <socios>...</socios>   <prestamos>...</prestamos> </biblioteca>
LibroType @id = L[0-9]+ precio@moneda = EUR

libros / libro

Cada libro tiene atributos y elementos obligatorios. Aqui estan los datos que mas suelen aparecer en practicas y consultas.

Campos del libro
@id, titulo, autor, anio genero, paginas precio[@moneda], disponible
PrestamoType keyref fechaDevolucion opcional

socios y prestamos

Los prestamos no guardan el objeto entero, guardan referencias. XSD comprueba que socioRef y libroRef apunten a ids existentes.

Relacion entre bloques
socio @id="S1" <-- socioRef libro @id="L2" <-- libroRef estado = ABIERTO o CERRADO

Conexiones del dataset

La gracia del curso no esta solo en leer etiquetas. Esta en entender la integridad del documento y luego hacer preguntas potentes con XPath.

libro @id se declara una sola vez -> prestamo/libroRef debe existir keyref
socio @id identifica al socio -> prestamo/socioRef debe existir keyref

Ruta de estudio

Esta pagina no cambia el contenido del curso. Lo ordena mejor para que el alumno entienda primero el flujo y luego practique.

UT1 · XML

Primero se trabaja la diferencia entre bien formado y valido. El alumno debe dejar de ver XML como "texto raro" y empezar a verlo como estructura de datos.

1
Bien formado

Etiquetas cerradas, un solo root, comillas correctas y anidacion valida.

2
Elementos y atributos

Los datos principales van en elementos; ids y metadatos breves suelen ir como atributos.

UT2 · XPath

Aqui aparece la parte mas operativa. El alumno hace preguntas al XML y aprende a devolver nodos, texto, filtros y funciones.

1
Rutas base

/, //, @ y [...] son la columna vertebral.

2
Funciones

count(), sum(), contains() y filtros combinados.

UT3 · XSD

El esquema impone tipos, secuencias, patrones, enumeraciones e integridad referencial. Aqui el curso pasa de "leer XML" a "validar reglas".

1
Tipos y reglas

xs:element, xs:complexType, xs:simpleType y restricciones.

2
Integridad

xs:key y xs:keyref para forzar ids unicos y referencias existentes.

Laboratorio interactivo

Aqui esta la parte fuerte del complemento: editor XML, evaluacion XPath real en navegador y analisis guiado de errores basados en las reglas del XSD del curso.

Controles del laboratorio

Carga el XML valido, el XML invalido o una version limpia. Despues prueba consultas XPath o ejecuta el analizador didactico del esquema.

  • XPath real: usa document.evaluate() del navegador.
  • Analisis XSD guiado: no sustituye a un validador real, pero reproduce las reglas clave del curso.
  • Ideal para clase: cambia el XML y muestra al momento que se rompe y por que.

    Radiografia del XSD

    El curso ya trae un esquema bastante decente para 1 DAM. Aqui lo tienes descompuesto visualmente para que no se vea como un bloque enorme de etiquetas.

    Tipos simples pattern enumeration

    Ids y enums

    El XSD define tipos para los identificadores y para varios valores cerrados. Esta es una de las mejores partes del curso porque enseña reglas utiles de verdad.

    <xs:simpleType name="IdLibroType"> <xs:restriction base="xs:string"> <xs:pattern value="L[0-9]+"/> </xs:restriction> </xs:simpleType> <xs:simpleType name="SiNoType"> <xs:restriction base="xs:string"> <xs:enumeration value="SI"/> <xs:enumeration value="NO"/> </xs:restriction> </xs:simpleType>
    Tipos complejos sequence atributos

    LibroType y PrestamoType

    La secuencia es obligatoria. Si el XML cambia el orden de hijos, el validador puede fallar aunque las etiquetas existan.

    <xs:complexType name="PrestamoType"> <xs:sequence> <xs:element name="socioRef" type="IdSocioType"/> <xs:element name="libroRef" type="IdLibroType"/> <xs:element name="fechaPrestamo" type="xs:date"/> <xs:element name="fechaDevolucion" type="xs:date" minOccurs="0"/> <xs:element name="estado" type="EstadoPrestamoType"/> </xs:sequence> </xs:complexType>
    Integridad xs:key xs:keyref

    La parte mas potente

    Este punto sube el nivel del curso: no solo hay estructura, tambien hay integridad referencial. Un prestamo no puede apuntar a un socio o libro inexistente.

    <xs:keyref name="PrestamoSocioRef" refer="SocioIdKey"> <xs:selector xpath="prestamos/prestamo"/> <xs:field xpath="socioRef"/> </xs:keyref> <xs:keyref name="PrestamoLibroRef" refer="LibroIdKey"> <xs:selector xpath="prestamos/prestamo"/> <xs:field xpath="libroRef"/> </xs:keyref>

    Errores tipicos que el alumno debe ver

    • Bien formado: si falta un cierre o hay comillas rotas, el XML ni se parsea.
    • Enumeracion: QUIZAS falla porque disponible solo admite SI o NO.
    • Orden: con xs:sequence no basta con "tener las etiquetas"; importa en que orden aparecen.
    • Referencias: S999 o L999 rompen la integridad.

    Ampliaciones buenas para examen

    • Limitar genero con enumeraciones.
    • Restringir anio con rango 1900-2100.
    • Forzar precio >= 0.
    • Definir un patron de 9 digitos para telefono.

    Practicas y proyecto

    El contenido ya existia en el curso. Aqui se reorganiza para que sea mas facil de impartir y mas facil de repasar antes de examen.

    Bloque XPath

    Secuencia recomendada para clase: empezar por seleccion simple, subir a filtros y cerrar con funciones.

    • //libro
    • //libro/titulo
    • //libro[genero='Fantasia']
    • //libro[paginas > 200]/titulo
    • //socio[not(telefono)]
    • count(//prestamo[estado='ABIERTO'])
    Como sacarle partido en clase

    Haz primero la ruta larga, luego la corta. Ejemplo: /biblioteca/libros/libro y despues //libro. Asi el alumno entiende de donde sale el atajo.

    Bloque XSD

    La practica fuerte del curso es abrir el XML invalido, validarlo y corregirlo hasta dejarlo listo.

    • Corregir disponible = QUIZAS.
    • Arreglar socioRef = S999.
    • Arreglar libroRef = L999.
    • Revisar si el root enlaza bien el esquema.
    Idea pedagogica

    Este ejercicio enseña muy bien la diferencia entre error estructural y error de reglas. Es uno de los mejores puntos del curso para clase practica.

    Proyecto final

    El cierre natural del curso es demostrar el flujo completo: corregir, ampliar y consultar.

    • Crear biblioteca_corregida.xml.
    • Anadir un libro nuevo con id valido.
    • Anadir un socio nuevo con id valido.
    • Anadir un prestamo que apunte al socio y libro nuevos.
    • Resolver las 12 consultas obligatorias.

    Checklist de dominio

    Si el alumno puede hacer esto sin copiar, va bien para 1 DAM.

    • Escribir un XML bien formado desde cero.
    • Explicar la diferencia entre bien formado y valido.
    • Detectar por que falla una enumeracion.
    • Entender un keyref roto.
    • Resolver 10-15 consultas XPath con filtros y funciones.
